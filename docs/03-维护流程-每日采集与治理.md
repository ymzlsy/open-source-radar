# 维护流程 · 每日采集与治理

> 更新机制：全自动定时任务（已选）。cron 负责「捞进来」，人工负责「晋升/复刻」。

## 每日采集 pipeline（cron 执行）

```
1. 抓取   → GitHub Trending(daily, 语言过滤) + 关注开发者的 GitHub/X 新动态 + AI builder 圈热门
2. 落原始 → 写 data/feed/raw/YYYY-MM-DD/*.json（只追加）
3. 去重   → 对比已有 projects.json/articles.json 的 id，过滤重复
4. 标准化 → 对齐 schema 字段，新条目 status=feed，写一句 why_interesting/why_useful
5. 趋势   → 已有项目 append 当日 stars 到 stars_history；开发者 append dynamics
6. 部署   → git add data/ && commit && push → Cloudflare Pages 自动构建上线
```

> 数据采集参考 Zara 的 `follow-builders`（监控 X/YouTube builder + remix 摘要），架构高度同构，可直接借鉴。

## 运行载体（已定，2026-06-10）

**GitHub Actions 云端 cron**：`.github/workflows/daily-collect.yml`，每日 UTC 09:00（北京 17:00）跑 `scripts/collect-followbuilders.mjs`，数据有变更则自动 commit + push → Cloudflare Pages 自动部署。不依赖本机开机。手动触发：仓库 Actions 页 workflow_dispatch。

> 注意：bot 会向 main 提交数据。本地改动前先 `git pull`。

## 信息源清单（2026-06-10 更新）

| 模块 | 主源 | 备源 |
|------|------|------|
| 3 项目 | GitHub Trending（日榜+周榜） | X 上的 repo 安利、Hacker News |
| 4 文章 | **follow-builders 中央 feed**（podcasts+blogs，MIT，零成本消费） | Hacker News、技术周刊 |
| 1 动态 | **follow-builders feed-x**（命中已关注开发者自动进 dynamics） | browser-act stealth-extract 兜底 feed 未覆盖的人（如 0xajc） |

### follow-builders 接入说明

- 上游：`raw.githubusercontent.com/zarazhangrui/follow-builders/main/feed-{x,podcasts,blogs}.json`，Zara 中心化抓取（X 官方 API + Supadata），每日更新
- 转换：`scripts/collect-followbuilders.mjs`——podcasts/blogs → articles（status=feed）；feed-x 按 handle 匹配 `data/developers/` 的 X 阵地，高赞前 2 条进 dynamics；原始快照存 `data/feed/raw/<日期>/`
- 风险控制：`generatedAt` 超 48h 自动跳过该源（依赖她停更不致脏数据）；所有条目标 `source: "follow-builders"` 可追溯
- X 抓取兜底：feed 没覆盖的人用 `browser-act stealth-extract`（免登录抓 X 公开页，已实测）

## 人工治理动作（我每隔几天做）

1. **扫 feed**：浏览模块3/4 新进 feed 条目。
2. **晋升**：有用的 → status 改 shortlisted/saved；作者值得追 → 建 developer 档（candidate→tracking）并双向回填 id。
3. **决策**：给 developer/project 填 `replicate_intent`。
4. **清理**：无关的 → dismissed。
5. **补全草稿档**：像 0xajc 这种，补 bio/动态后转 tracking。

## 数据质量规则

- 每个新 feed 条目至少有一句 `why_*`，否则不入库（防噪声）。
- 同一 repo 只一条记录，靠 `id=owner__repo` 去重。
- raw 层永不删，标准层可改。
- 半年后若 JSON 过大 → 迁 SQLite/D1，schema 不变。

## 阶段性复盘（每月）

- star 趋势：谁在真起势、谁是昙花。
- 我的 replicate_intent=build/fork 的项目，实际动手了几个？（项目的真正 KPI）
- 关注列表健康度：tracking 里有没有该 archive 的。

## AI 协作约定

后续「喂数据」时的标准动作：判断新增/更新/退役/纠错/补图 → 抽字段 → 更新单档与总表 → 必要时回写 secondbrain（`wiki/projects/open-source-radar/`）。
