# open-source.karaithy.com · 价值与边界研究

> 建档 2026-06-09 ｜ 类型：长期积累型数据项目 ｜ 对外属性：公开作品集

## 一句话定义

这是一个围绕「我关注的个人开发者及其项目」的长期数据项目，目标是把零散的关注/灵感沉淀成结构化档案与每日情报流，最终驱动我自己去**实践→复刻→找到想做的项目→商业化落地**，以一个独立 React 网站对外公开呈现。

## 为什么值得做

1. **从被动刷到主动选**：我现在对开发者/项目的关注是碎片化的、刷过即忘。结构化后能横向对比、判断「哪些对我有用」。
2. **决策工具属性**：核心不是收藏，而是 `replicate_intent`（none/watch/fork/build）这个字段——逼着每条记录回答「我要不要动手」。
3. **复利资产**：每日采集 + star 趋势快照，时间越长越值钱；半年后能看出谁在真正起势。
4. **个人品牌副产物**：公开在 karaithy 子域名，本身就是「我在认真研究 indie 生态」的对外证据。

## 核心价值锚点

| 模块 | 本质问题 | 关键产出 |
|------|----------|----------|
| 1 我的开发者 | 谁值得我持续追、能借鉴什么 | 深度档案 + 商业化分析 + 复刻意向 |
| 2 推荐开发者 | 还有谁我该认识、他的阵地在哪 | 轻量卡片 + 可一键晋升到模块1 |
| 3 热门项目流 | 最近什么火、能不能拉来改 | 每日 feed + star 趋势 + 复刻意向 |
| 4 热门文章流 | 最近什么值得学 | 每日 feed + 收藏/已读状态 |

## 边界：不做什么

- **不做大而全的 trending 聚合站**：只收「对我有用」的，宁缺毋滥。GitHub Trending 全量不是目标。
- **不做社交/评论**：纯个人决策工具 + 作品集，无 UGC。
- **不把私人毒舌评价上线**：公开站点 `my_take` 保持客观克制；真正直白的判断只存本地，不进部署数据。
- **不在 MVP 阶段做账号体系/后台**：数据即 JSON 文件，cron 写、git 部署，零后端。
- **不追求实时**：每日一次采集足够。

## 最关键的 5 个设计判断

1. **项目与开发者解耦但可关联**：`project.author_developer_id` 把模块3 的项目挂到模块1 的人。看到好项目→一键把作者纳入关注，这是你明确要的流转。
2. **四类实体共用「生命周期 status」**：开发者 candidate→tracking→archived；项目 feed→shortlisted→replicated；文章 feed→saved→consumed。避免越积越乱。
3. **原始层/标准层/分析层三分**：cron 抓的原始快照存 `data/feed/raw/`，去重标准化后进 `projects.json/articles.json`，人工深度分析单独存。
4. **模糊值统一用 null + 待补充**：抓不到的（如 0xajc）建草稿档而非跳过，保留线索。
5. **全自动写入但默认 status=feed**：cron 只负责「捞进来」，晋升/收藏/复刻是你的人工决策动作。

## 待验证项

- X 数据采集的合规与可行性（主页需登录态，nitter 不稳）——可能要降级为「我手动喂 X 链接 + cron 只管 GitHub」。
- 全自动 cron 的运行载体：本机 scheduled-task 还是云端 cron？影响是否需要常开机。
- star 趋势快照的存储增长——单项目每日一条，半年 ~180 条/项目，JSON 仍可控；超量再迁 SQLite。
