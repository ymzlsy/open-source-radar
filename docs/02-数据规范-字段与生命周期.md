# 数据规范 · 字段与生命周期

> 权威定义见 `data/schema/*.json`，本文是人读版说明与约定。

## 四类实体一览

| 实体 | 文件 | schema | 模块 |
|------|------|--------|------|
| Developer 开发者 | `data/developers/<id>.json`（一人一档） | developer.schema.json | 1 |
| Recommended 推荐开发者 | `data/recommended-developers.json`（单数组） | 见下 | 2 |
| Project 项目 | `data/feed/projects.json`（单数组） | project.schema.json | 3 |
| Article 文章 | `data/feed/articles.json`（单数组） | article.schema.json | 4 |

## 三层数据结构

- **原始层 raw**：`data/feed/raw/YYYY-MM-DD/*.json`——cron 当日抓取的原始快照，只追加不修改，可追溯。
- **标准层 standard**：`projects.json` / `articles.json` / `developers/*`——去重、标准化、字段对齐后的可用数据，前端直接读。
- **分析层 analysis**：写在标准层记录的 `analysis` / `my_take` / `commercialization` 字段里，人工或 AI 后补。

## 生命周期（状态机）

```
Developer:    candidate ──→ tracking ──→ archived
Project:      feed ──→ shortlisted ──→ replicated
                  └────────────────→ dismissed
Article:      feed ──→ saved ──→ consumed
                  └──────────→ dismissed
Recommended:  promoted=false ──(晋升)──→ 生成 Developer(tracking)，回填 promoted_to_developer_id
```

- **candidate / feed**：cron 自动捞进来的默认态，未经我决策。
- **tracking / shortlisted / saved**：我主动标记「要持续追/收藏」。
- **replicated / consumed**：已落地行动（复刻了/读完了）。
- **archived / dismissed**：明确不再关注。

## 复刻意向 replicate_intent（决策核心字段）

`none`（不动） / `watch`（观望） / `fork`（直接拉来改） / `build`（借鉴自建）。
这是整个项目的「行动出口」——每条 developer/project 都该回答它。

## 关联关系

- `project.author_developer_id` → 关联模块1 开发者；为 null 表示作者尚未纳入关注。
- `developer.project_ids[]` → 反向引用其代表项目。
- `article.author_developer_id` → 同上。
- 看到模块3 好项目 → 把作者建成 developer → 回填两端 id，完成「项目流→关注列表」的流转。

## 字段约定

- 日期一律 `YYYY-MM-DD` 字符串。
- 未知数值用 `null`，不要用 0 或留空字符串混淆。
- `id` slug：开发者用 handle；项目用 `owner__repo`；文章用语义 slug。
- 公开站点的 `my_take` 保持客观克制；直白判断只存本地草稿、不进部署。
- 抓不到的对象建**草稿档**（status=candidate + summary 写明缺口），不要直接丢弃线索（参考 `0xajc.json`）。

## 推荐开发者字段（模块2，轻量）

`id / name / handle / oneliner / bases[] / tags / why_recommended / related_to / promoted / promoted_to_developer_id / discovered_at / source`。
比模块1 少了 story/commercialization/dynamics——推荐池只需「快速判断 + 阵地」，晋升后才补全。
